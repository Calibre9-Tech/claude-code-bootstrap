## Verification Gate

**Iron Law: NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE**

Before claiming any task is done, fixed, or passing:
1. IDENTIFY what command proves the claim
2. RUN it (fresh and complete)
3. READ the full output and check exit code
4. VERIFY output actually confirms the claim
5. ONLY THEN make the claim, and include the output

| Claim | Required evidence |
|-------|-----------------|
| Tests pass | `pytest` output: N passed, 0 failed |
| Linter clean | `ruff check .` output: All checks passed |
| Migration applied | `alembic upgrade head` ran, query confirms schema |
| API endpoint works | `curl` or httpx output showing correct response |

Banned phrases without fresh evidence: "should work", "try it now", "looks correct", "should be fixed".

## FastAPI Conventions

- Routes in `app/routers/` — one file per resource
- Pydantic models in `app/schemas/` — separate request/response schemas
- Business logic in `app/services/` — not in route handlers
- DB sessions via dependency injection (`Depends(get_db)`)
- All endpoints require auth unless explicitly marked public

## Database Conventions

- Alembic for all migrations: `alembic revision --autogenerate -m "description"`
- SQLAlchemy models in `app/models/`
- Always use parameterized queries — never f-strings in SQL
- Add indexes on all foreign keys and frequently-filtered columns
